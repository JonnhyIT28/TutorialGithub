REPOSITORIO LOCAL

1- CHEQUEAR QUE GIT ESTE INSTALADO

git --version

2- CONFIGURAR GIT

git config --global user.name "Pepito Valenciaga"

git config --global user.email "pepilo.valenciaga@gmail.com"

3- CREAR UNA CARPETA PARA EL PROYECTO

mkdir nombredelproyecto
(*mkdir crea un nuevo directorio*)

cd nombredelproyecto
(cd cambia el enfoque a este directorio de trabajo)

SI YA TENEMOS UNA CARPETA ARMADA, SOLO LE HACEMOS CLICK DERECHO Y PONEMOS "Git Bash Here"

4- INICIAR GIT

git init
(se crea el repositorio git)

5 - AGREGAR ARCHIVOS UNTRACKED

Los agregamos desde el editor de texto de preferencia.

6 - STATUS (A-DETECCION DE FILES)

ls
(ls arma una lista de todos los archivos del directorio)

luego podemos chequear el status de Git para ver si nuestro archivo esta siendo detectado por git:

git status

los diferentes status de archivos son:
Tracked - files that Git knows about and are added to the repository

Untracked - files that are in your working directory, but not added to the repository

7 - STAGING (B-TRACKEO DE MODIFICACION DE ARCHIVOS, ANTES DE HACERLES COMMIT)

git add nombredelarchivo.html



EL ARCHIVO DEBERIA ESTAR EN EL STAGING ENVIRONMENT, VAMOS A CHEQUEAR EL STATUS

git status

PARA AGREGAR MAS DE UN ARCHIVO A LA VEZ:

git add --all

CHEQUEAMOS EL STATUS Y LISTO

git status

EL SHORTHAND SERIA:

git add -A

8 - COMMITS (C-PUNTO DE GUARDADO EN EL REPO LOCAL, EN EL CUAL SE PUEDE VOLVER EN CASO DE ENCONTRAR UN BUG O QUERER HACER ALGUN CAMBIO, SIEMPRE SE DEBE ENVIAR CON UN MENSAJE)

git commit -m "Primer commit del machete Git"

SI NECESITAMOS HACER UN CAMBIO MUY PEQUEÃ‘O SIN NECESIDAD DE HACER STAGE. PRIMERO HACEMOS LA MODIFICACION, LUEGO CHEQUEAMOS EL STATUS:

git status --short

LOS STATUS SHORT PUEDEN SER LOS SIGUIENTES:

?? - Untracked files A - Files added to stage M - Modified files D - Deleted files

EL ARCHIVO SE VERA "M" POR Modified, ASI QUE DIRECTAMENTE HAGAMOS EL COMMIT

git commit -a -m "le agrege mas h1 al machete"

PARA VER EL HISTORIAL DE COMMITS DEL REPOSITORIO:

git log

9 - HELP

EN CASO DE OLVIDAR COMANDOS, USAMOS HELP

git commmandoquequeeramosinfoo -help o sino git help --all

10 - BRANCH (D- ES UNA NUEVA VERSION SEPARADA DEL REPOSITORIO PRINCIPAL, LA CUAL SE MODIFICA PARA LUEGO AGREGARLA AL REPO PRICIPAL)

git branch nombre-del-branch-ejemplo-imagenes-machetito

AHORA DEBEMOS CONFIRMAR QUE FUE CREADO:

git branch

NOS APARECERAN LOS BRANCHS CREAODS, Y ABJO UN * CON EL BRANCH DONDE ESTAMOS UBICACOS. PARA CAMBIAR DE BRANCH HACEMOS:

git checkout nombre-del-branch-ejemplo-imagenes-machetito

NOS DARA UN MENSAJE DE QUE NOS MOVIMOS DE BRANCH

AL HACER UNAS MODIFICACIONES EN LOS FILES, PODEMOS HACER STAGING Y COMMITS EN LOS FILES PARALELOS:

git add --all

CAMBIO DE BRANCHS

git checkout master

CON ESTO VOLVEMOS AL BRANCH PRINCIPAL, Y AL CODIGO ANTES DE SER MODIFICADO. LO MISMO OCURRIRA CON FILES, SI AGREGAMOS FILES EN EL BRANCH MACHETITO, NO ESTARAN EN EL MASTER AL VOLVER.

BRANCH DE EMERGENCIA

EN CASO DE QUE OCURRA UN BUG EN MASTER, PERO NO HAYAMOS TERMINADO CON MACHETITO, Y DEBEMOS SOLUCIOANR EL PROBLEMA, CREAMOS UN BRANCH DE EMERGENCIA

git checkout -b branch-de-emergencia-ejemplo

SOLUCIONAMOS EL ERROR, HACEMOS STAGING, COMMIT, Y YA PODEMOS UNIR LOS 2 BRANCHS

LIGAR BRANCHS

PRIMERO DEBEMOS IR AL MASTER BRANCH

git checkout master

AHORA LOS PODEMOS UNIR

git merge branch-emergencia

COMO EL MASTER NO FUE MODIFICADO, GIT VE AL BRANCH DE EMERGENCIA COMO UNA EXTENSION DEL MASTER, Y LO SOBREESCRIBEE A TRAVES DEL MERGE. AHORA PODEMOS ELIMINAR EL BRANCH DE EMERGENCIA

git branch -d branch-emergencia

AHORA VAMOS A LIGAR LAS MODIFICACIONES DE MACHETITO

git checkout machetito

SUPONINOD QE MACHTITO ESTA COMPLETO, Y YA HICIMOS STAGE Y COMMITS , VOLVEMOS AL MASTER CON CHECKOUT PERO Y AL HACER MERGE DARA "CONFLICT" EN STATUS NO SALDRA QUE TENEMOS "UNMERGED PATHS"

LUEGO NOS MOSTRARA EN EL ARCHIVO LAS DIFERENCIAS ENTRE LOS FILES Y PODREMOS EDITARLO COMO QUERAMOS

LUEGO LE HACEMOS STAGING Y EL CONFLICTO ESTARA SOLUCIONADO PARA PODER HACER EL COMMIT

YA PODEMOS ELIMINAR MACHETITO CON BRANCH -d

REPOSITORIO VIRTUAL

1 - CONECTAR AL REPO VIRTUAL

DESPUES DE CREAR UN REPOSITORIO EN GITHUB, COPIAMOS EL LINK Y LO PEGAMOS EN EL SIGUIENE COMANDO:

git remote add origin https://github.com/w3schools-test/hello-world.git

AHORA VAMOS A LIGAR EL MASTER BRANCH A LA URL Y SETEARLO COMO EL BRANCH REMOTO POR DEFECTO:

git push --set-upstream origin master

2 - EDITAR CODIGO EN GITHUB

APRETAMOS EL ARCHIVO QEU QUEREMOS EDITAR, O APRERTAMOS EL LAPIZ PARA EDITAR TEXTO, LUEGO HACEMOS UN COMMIT Y CLICK EN :

Commit directly to the master branch

AHI HACEMOS UN COMMIT EN EL MASTER BRANCH

3 - PULL DE GITHUB

PULL PERMITE MANTENERTE AL DIA CON LA COPIA MAS ACTUAL DEL PROYECTO, ESTA COMPUESTO POR fetch y merge:

fetch SINCRONIZA EL HISTORIAL DE CAMBIOS DEL REPO VIRTUAL CON EL LOCAL, HACIENDO:

git fetch origin

UNA VEZ QUE EL HISTORIAL SE ACTUALIZE, PODEMOS CHEQUEAR EL STATUS:

git status

NOS SALDRA QUE NUESTRO BRANCH ESTA DETRAS EL "ORIGIN/MASTER" POR 1 COMMIT, QUE SERIA EL README QUE EDITAMOS RECIEN

PODEMOS VERLO HACIENDO UN LOG, AHORA SERA:

git log origin/master

ESTE NOS MOSTRARA EL COMIT HECHO EN EL REPO VIRTUAL

TAMBIEN PODEMOS VERFICIARLO EN VIENDO LAS DIFERENCIAS ENTRE EL MASTER Y EL ORIGIN/MASTER

git diff origin/master

NOS APARECERA EN ROJO LAS DIFERENCIAS, AHORA PODEMOS HACER merge
